# 知曜 Agent 架构 × RAG 迭代方向报告

> 2026-05-31 · 基于逐文件代码调研 + 本会话写入侧地基实现 · 战略级技术评估

## 摘要

知曜内置 Agent 已是**功能完整的 MVP**：编排层（三推理模式）、能力层（17 工具）、认知层（情节记忆 + RAG + 用户画像）、基础设施层（pgvector + BGE-M3 + Celery）都是真实闭环代码。**但作为"学习操作系统的认知中枢"，它当前更像一个'会用工具的通用知识问答 Agent'，而非'与学习场景深度联动的专家 Agent'。** 最大的结构性断层有两个：(1) **RAG 只索引"知识内容"（笔记/KP/章节），完全不索引"学习行为与项目数据"（错题/训练/项目/考试/闪卡）**——RAG 与产品核心场景没有真正联动；(2) **上下文管理是按消息条数粗暴截断**，无 token 预算、无长期摘要，长对话必然丢上下文。本会话已补完 RAG **写入侧地基**（消除"读通写断"），这是后续一切联动的前提，但联动本身尚未开始。

## 一、五层架构现状解读

> 每层标注：**定位 / 已搭建（实证）/ 缺口 / 严重度**

### L1 应用层（用户触点）
- **定位**：用户与 Agent 的交互界面与触发入口。
- **已搭建**：SSE 流式对话（thinking/delta/error/done 事件齐全）；StudySpace / 模考专属模式；图片上传转文字；TTS 语音输出；对话历史浏览 + ILIKE 搜索。
- **缺口**：① **工具调用过程不可视**（C-13 未做）——用户看不到 Agent 在"查知识库/建项目/出题"，黑箱感强，信任度低；② Agent 状态机（thinking/planning/executing）后端有，**前端无进度流可见**（C-14）；③ 引用展示弱——RAG 注入了 top-5 但前端不展示"答案来自你哪篇笔记"，可溯源性差。
- **严重度**：🟠 中（影响信任与体验，不影响正确性）。

### L2 编排层（推理与流程编排）
- **定位**：决定"一句话怎么拆解、走哪条推理路径、调哪些工具"。
- **已搭建**：复杂度分类器（关键词 fast-path + LLM 兜底）；**双推理模式**——简单走 ReAct（≤5 轮工具循环），复杂走 Plan-Execute-Verify-Reflect（≤2 reflect 轮，工具名白名单 sanitize）；管线含 PII mask → 内容审核 → 上下文加载 → RAG/记忆注入 → 工具循环 → 流式回答。
- **缺口**：① **无显式多意图拆分**——一条消息只走一条复杂度路径，多意图靠 LLM 单轮自行处理，易漏；② Plan-Execute 的 `execute` **纯顺序、步间无数据传递**（前一步结果喂不进后一步参数），多步依赖任务无法真正串联；③ `verify` 失败时静默当成功（"treat as ok"），自检不可靠时无信号；④ 复杂分支跑完 Plan 后直接跳过 ReAct，若 plan 全失败仍可能基于失败结果作答。
- **严重度**：🟠 中（推理深度的天花板，影响复杂任务质量）。

### L3 能力层（工具 / Action）
- **定位**：Agent 能"做"的事——查、建、改、算。
- **已搭建**：**17 个工具**全部真实实现（非占位），覆盖上下文/诊断/计划/任务/KP/训练/考试/笔记/项目/记忆/RAG 召回/费曼/随堂测验；每次调用 `_record_trace` 写 tool trace。
- **缺口**：① `plan_study_schedule` 是**模板化 hardcode**（固定标题/45min/取模分配日期），不是 LLM 真规划——"自主规划"的核心卖点是空的；② tool_trace 的 **token/cost/was_helpful/user_action 字段全部留空**——成本核算与"哪个工具有用"的反馈维度是空壳，无法做工具级优化；③ 工具无"执行预览/确认"机制，高风险写操作（建项目/批量建 KP）直接执行。
- **严重度**：🟠 中（规划质量 + 可观测性）。

### L4 认知层（记忆 / 检索 / 上下文）⭐ 报告重点
- **定位**：Agent 的"长期记忆 + 知识检索 + 当前上下文"——决定它"懂不懂这个用户、调不调得到对的知识"。
- **已搭建**：
  - **情节记忆**（最成熟）：`agent_episodes` + 8 处真实写入点（checkin/复习/费曼/StudySpace/项目/周报 + beat 扫描 inactive/exam）+ 90 天保留 + importance 加权检索。
  - **用户画像记忆**：`user.agent_memory`（Agent 主动 save_memory）。
  - **RAG 语义检索**：pgvector HNSW + BGE-M3（1024 维）+ user 隔离（me OR official）+ 每条消息自动注入 top-5 + `retrieve_knowledge` 工具主动召回。
  - **写入侧地基（本会话补全）**：KP/note 的 create/update/delete + Agent 建 KP 全部接入向量维护，消除"读通写断"。
- **缺口（这是全局最大短板）**：
  1. **RAG 只覆盖 4 类 doc_kind：kp / note / chapter / episode**——全是"知识/记忆"。**错题、训练记录、闪卡、项目结构、考试**完全不进向量库。
  2. **RAG 与场景零联动**：训练出题、错题归因、项目规划、模考生成时**不检索**用户相关历史。RAG 只在"聊天"时被动注入。
  3. **上下文管理粗糙**：Redis 历史**按消息条数截断（最近 20 条 / 24h TTL）**，无 token 预算、无长对话摘要、无跨会话长期上下文。长消息撑爆窗口，旧上下文 24h 后蒸发。
  4. **official 内容信息量低**：章节只 embed "科目·章节·课时"三段标题，无正文，召回质量天花板低。
- **严重度**：🔴 高（这是 Agent 从"通用问答"升级为"学习专家"的关键瓶颈）。

### L5 基础设施层
- **定位**：LLM / 嵌入 / 向量库 / 异步 / 可观测性的底座。
- **已搭建**：DeepSeek V4 Flash 主 LLM；BGE-M3 本地嵌入；pgvector pg17 + HNSW cosine；Celery（9 任务 + beat）；**F 系列加固已完成**——LLM 速率限制、配额（DB 权威 + Redis）、死信队列（F-11）、文件鉴权（F-08）、magic bytes。
- **缺口**：① **嵌入 CPU 同步**——BGE-M3 在 `to_thread` 跑，高并发吞吐受限，episode 写入还同步阻塞调用方；冷启下 1.3GB 模型；② **可观测性空洞**——RAG 无召回质量埋点（命中率/score 分布/是否被引用），token/cost 未落账；③ official 向量靠手动脚本回填（backfill 端点本会话刚补，但全量首灌仍需触发）。
- **严重度**：🟡 中低（规模化前可接受，规模化后成本/性能/可观测性会卡脖子）。

## 二、RAG × 上下文 闭环诊断（企业/个人产品视角）

把"闭环"拆成四个维度，逐个判定通断：

| 维度 | 状态 | 说明 |
|---|---|---|
| **读路径**（检索→注入→生成） | ✅ 通 | 向量检索 + user 隔离 + 自动注入 top-5 + 引用提示，read path 完整 |
| **写路径**（数据→索引→可检索） | ✅ 通（本会话补全） | KP/note/Agent 建 KP 全接入；失效重建闭环；backfill 入口 |
| **上下文窗口**（短期记忆管理） | ⚠️ 半通 | 仅条数截断，无 token 预算/摘要/长期上下文，长对话丢信息 |
| **场景联动**（RAG↔核心业务） | ❌ 断 | RAG 只服务"聊天"，不服务训练/错题/项目/考试；行为数据不入库 |

**结论**：作为"能回答学习问题的 Agent"，闭环基本打通（读写都通了）；但作为"**学习操作系统的认知中枢**"，**场景联动这一环是断的**——这正是你强调的"只打通没用，必须和重要使用场景联动"。

## 三、RAG 现状：方向 + 已搭建内容

**当前方向定性**：**"用户私有知识 + 官方课程内容的语义检索增强问答"**——即把用户的笔记/知识点/章节向量化，在 Agent 回答时召回相关片段注入 prompt，让回答"基于用户自己的学习材料"。这是 **RAG 1.0：知识库问答**。

**已搭建（实证）**：
- 向量库：`document_embeddings`（pgvector Vector(1024) + HNSW cosine + 唯一约束 doc_kind+doc_id+chunk+model）
- 嵌入：BGE-M3 本地，1024 维归一化，`asyncio.to_thread` 防阻塞，lazy 单例
- 切块：按 `##/###` 标题 + 1200 字硬切；note 拆 exam_version/full_version
- 隔离：`user_id=me OR (official AND user_id IS NULL)`，subject/project/kind 可选过滤
- 注入：每条消息自动 `format_for_prompt` 拼 top-5 + `retrieve_knowledge` 工具
- **写入侧地基（本会话）**：`rag_index` 统一入口，KP/note CRUD + Agent 建 KP 全维护 + backfill

## 四、核心断层：RAG 没有和项目使用场景联动

这是报告最关键的判断。当前 RAG 是"**旁挂式知识问答**"，不是"**嵌入式场景智能**"。具体表现：

**(A) 数据覆盖断层**——RAG 只看得见"用户写了什么知识"，看不见"用户学得怎么样":
- ❌ 错题（mistakes）不入库 → Agent 无法"基于你过去同类错题"分析
- ❌ 训练记录（training）不入库 → 出题时无法"避开你已掌握、强化你薄弱"
- ❌ 项目结构（project/tree）不入库 → 规划时无法"基于你项目当前进度"
- ❌ 考试（exams）不入库 → 备考建议无法"结合你的考试时间线 + 知识盲区"

**(B) 场景调用断层**——RAG 只在"聊天"被动触发，不在业务动作中主动调用：
- 训练出题（`compose-quiz`）、错题归因、项目树生成、模考生成——这些**最该用 RAG 的场景，都不检索**。

**这意味着什么**：知曜的差异化定位是"AI 学习操作系统"。如果 RAG 只能回答"什么是导数"，那和通用大模型无差别；只有当 RAG 能回答"**结合你上周这道导数错题、你的高考时间线、你项目里这一章的进度，你现在该练什么**"——RAG 才真正成为产品护城河。当前距离后者还差**场景联动**这一大步。

## 五、迭代方向（分层 + 每项的实际闭环影响）

> 按"地基→联动→质量→规模"四阶段。每项标注**在实际使用闭环中产生的有效影响**。

### 阶段 0（已完成）：写入侧地基
- ✅ 本会话完成。**影响**：RAG 召回的数据不再不完整/过时——所有后续迭代的前提。

### 阶段 1（建议优先）：RAG 场景数据联动 —— 把"学习行为"纳入认知
1. **扩展 doc_kind 覆盖**：错题/训练结果/项目节点入向量库（或结构化召回）。
   - **闭环影响**：Agent 回答从"知识正确"升级为"**针对这个用户当前状态正确**"——出题避重就轻、错题举一反三、规划贴合进度。这是从"问答机器人"到"学习专家"的质变。
2. **场景级 RAG 主动调用**：训练出题前检索"用户该 KP 的历史错题 + 掌握度"；项目树生成前检索"用户已有相关 KP/笔记"。
   - **闭环影响**：每个核心动作都"带着用户记忆做"，留存与效果显著提升；这是"学习操作系统"叙事的兑现。

### 阶段 2：上下文工程 —— 让 Agent 真正"记得住"
3. **token 预算化上下文**：用 token 计数替代条数截断 + 超限时对旧轮次做 LLM 摘要压缩。
   - **闭环影响**：长辅导对话不再"聊着聊着忘了前面"，连续多轮深度辅导成为可能。
4. **跨会话长期上下文**：把关键结论沉淀进 episodic/画像记忆，新会话自动召回。
   - **闭环影响**：用户每次打开不用"重新自我介绍"，Agent 持续"认识"用户——粘性核心。

### 阶段 3：召回质量
5. **official 章节正文 embed**（替代仅标题）+ **rerank**（cosine 召回后用 cross-encoder/LLM 重排）+ **引用强制校验**。
   - **闭环影响**：召回更准、答案可溯源（"来自你《XX》笔记第 3 段"），可信度与教学严谨性提升。

### 阶段 4：规模化与可观测
6. **嵌入服务化**（GPU 或云嵌入 API，按预算）+ **RAG 质量埋点**（命中率/score/是否被引用）+ tool_trace token/cost 落账。
   - **闭环影响**：支撑并发、看得见"RAG 到底有没有用"、成本可控可优化——从"能跑"到"可运营"。

## 六、企业 vs 个人产品差异化考量

| 维度 | 个人产品 | 企业/机构产品（如学校、教培）|
|---|---|---|
| 知识库 | 用户私有笔记为主 | + **机构私有题库/教材/教研资料**（多租户共享 official）|
| 隔离 | user 级（已实现）| + **组织/班级级**多租户隔离与权限 |
| 数据 | 隐私优先、可本地 | **审计留痕**（谁问了什么、引用了哪些资料）|
| 成本 | 对价格敏感，CPU 本地嵌入合理 | 可承担 GPU/云嵌入换吞吐 |
| RAG 重心 | "我的学习历史"联动 | "机构知识资产 + 学情数据"联动 |

**判断**：当前架构（user 级隔离 + official 共享）是**个人产品就绪、企业产品需补多租户层**。若要做企业，组织级隔离/权限/审计是必加项；个人产品则优先做阶段 1-2 的场景联动与上下文。

```
                         (RAG 护城河起点)        (留存核心)         (可信度)         (可运营)
```

- **强烈建议下一步走阶段 1（场景数据联动）**——这是 RAG 从"通用问答"变成"知曜专属护城河"的唯一路径，也是你说的"必须和重要使用场景联动"的直接落地。
- 阶段 1 与前端 C-12（文档上传 UI/引用展示）可并行：后端做联动、前端做可视，互不阻塞。
- **需用户决策**：①是否本期做企业多租户（影响隔离层设计）②嵌入性能升级时机（涉及月预算 ¥500-3000）③官方题库/教材是否纳入 RAG（涉及内容版权与采购）。
