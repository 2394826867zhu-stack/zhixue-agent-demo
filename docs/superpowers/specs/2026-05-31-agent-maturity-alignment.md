# 知曜 Agent 成熟度对齐与迭代规划优化

> 2026-05-31 · 以「Agent 优化迭代方向说明」8 维度成熟度框架为标尺，对齐知曜现状并优化迭代路线
> 一句话标尺：成熟 Agent 不是"更会说"，而是"搜得准、调得稳、错得少、查得到、改得动、接得住业务系统"

## 一、8 维度 × 知曜现状对齐（基于逐文件核验）

| # | 维度 | 知曜现状（实证）| 成熟度 | 关键 gap |
|---|---|---|---|---|
| 1 | **检索层** | 纯向量语义（pgvector cosine top-k）+ 部分元数据过滤（subject/project/kind）；写入侧地基 + 三级隔离已补 | 🔴 **25%** | 无 BM25/全文、无混合检索、无 rerank、无分问题类型策略、精确字段（KP名/章节/考试名）召回不稳 |
| 2 | **评估层** | 仅 `tests/eval/test_agent_quality.py` 10 个对话 case 骨架，但 `test_case` 是 `@skip+pass`**未实现**；**无任何检索指标** | 🔴 **8%** | 无 Recall@K/MRR/nDCG/引用准确率/工具成功率、无应召回文档标注、无 A/B、无持续评测 |
| 3 | **工具层** | 17 工具 + JSON schema（prompt 内）+ `tool_trace` + F-11 DLQ（任务级失败兜底）| 🟠 **50%** | 参数无防幻觉校验（ID/名称/金额）、无工具级结构化重试/降级、trace 的 token/cost/反馈字段空 |
| 4 | **架构层** | `agent_tools` 直连 service/DB（单层封装）| 🟠 **40%** | 无 MCP/标准工具协议；企业多系统（CRM/ERP/工单）接入时缺解耦 |
| 5 | **上下文层** | RAG top-5 + 记忆 top-3 + 历史 20 条**简单拼接**进 prompt；历史按**条数截断**（24h TTL）| 🟠 **35%** | 无压缩/去重/排序/证据标注、无 token 预算、无结构化状态管理、长对话丢上下文 |
| 6 | **可观测性** | `tool_trace`（latency/status）+ F-11 DLQ | 🟠 **35%** | RAG 无召回埋点（query/召回/rerank/引用）、trace token/cost 空、无错误分类统计 |
| 7 | **安全层** | user + org 三级隔离、PII mask、LLM 速率限制、文件鉴权、token 配额、agent regenerate/correct/undo | 🟢 **60%**（相对最成熟）| 读写工具未分级、高风险写操作无用户确认、无字段级权限脱敏 |
| 8 | **业务闭环** | 学习核心场景明确；RAG 与场景**联动断**（错题/训练/项目不入库、不主动召回）| 🟠 **40%** | 场景联动未通、失败案例未沉淀为评估样本 |

**整体定位**：知曜 Agent =「**能回答 + 基础设施加固扎实（安全层领先）**」，但「**可评估、检索精准、上下文可控、业务深度联动**」四大成熟支柱尚未建立——精确卡在框架所说的 **"Demo 能力 → 工程能力"过渡点**。

## 二、这份框架对原 RAG 规划的三处修正

1. **评估层被严重低估** —— 原规划是"功能驱动"（先做场景联动/检索质量）。框架正确指出"没有评估系统，Agent 优化就是凭感觉"。**评估层必须从"以后做"提为"前置底座"**。这是最大的规划修正。
2. **检索层不止 rerank** —— 原规划只提 rerank。框架强调**混合检索（BM25 + 语义 + 元数据精确过滤）+ 分问题类型策略**，尤其针对精确字段。知曜可用 PostgreSQL 原生全文(tsvector)/pg_trgm 实现 BM25 类检索，与 pgvector 混合，**无需引入新组件**。
3. **架构层 MCP 解耦是新增维度** —— 原规划未涉及。当前单体直连够用；企业多系统接入时通过 MCP/工具服务层解耦。

## 三、优化后的迭代路线（关键调整：度量先行）

| 阶段 | 对应维度 | 内容 | 状态 |
|---|---|---|---|
| **A 地基** | 检索写入 / 安全 | RAG 写入侧闭环 + 多租户隔离层 + 云嵌入选型 | ✅ **已完成** |
| **B 度量底座** 🎯 | **评估层(2)** | 激活 10 case 骨架 + 扩到覆盖（精确/语义/多轮/工具/业务判断）+ 标注应召回文档/证据/正确工具 + 实现 Recall@K·MRR·nDCG·引用准确率·工具成功率 + 可重复评测脚本 | **新前置** |
| **C 检索升级** | 检索层(1) | BM25(PG 全文/pg_trgm) + 语义 **混合检索** + 元数据精确过滤 + **rerank** + 分问题类型策略；**每步用 B 的评估集 A/B 量化提升** | |
| **D 上下文工程** | 上下文层(5) | 来源分区(用户/检索/工具/历史/规则) + 压缩去重排序 + 证据标注 + token 预算 + 结构化状态管理 | |
| **E 工具/可观测/安全 加固** | 工具(3)/可观测(6)/安全(7) | 参数防幻觉校验 + 工具级结构化重试/降级 + RAG 全链路埋点 + 错误分类 + 读写工具分级 + 高风险确认 + 字段脱敏 | |
| **F 业务联动** | 业务闭环(8) | 错题/训练/项目入 RAG + 场景级主动召回（出题/错题/规划时检索）+ **失败案例沉淀回 B 评估集** | |
| **G 架构解耦** | 架构层(4) | MCP/工具服务层（CRM/ERP/工单 → 标准工具），企业多系统接入时 | 中后期 |

**闭环飞轮**：B（度量）→ C/D/E/F（优化）→ F 把失败案例沉淀回 B → 评估集越来越全 → 优化越来越准。这正是框架说的"持续评估机制"。

## 四、关键调整说明（为什么这样优化）

- **为什么评估层(B)前置**：C/D/E/F 的每一项优化（混合检索、rerank、上下文压缩、场景召回）都需要"提升了多少"的度量，否则是盲改。知曜已有 10 case 骨架（仅未实现），**激活 + 补检索指标投入小、回报大**，是性价比最高的起点。
- **为什么检索层(C)排第二**：它是"搜得准"的核心，且 gap 最大（🔴 25%）；有了 B 的度量，C 的每步都能证明效果。
- **为什么安全层(7)不单列阶段**：知曜安全层已相对成熟（🟢 60%，F 系列 + 三级隔离），只需在 E 补"读写分级 + 高风险确认"——这是接入业务写操作前的必补项，不是从零建。
- **为什么架构层(G)垫底**：MCP 解耦的价值在多业务系统接入时兑现；当前单体 + agent_tools 单层封装够用，过早 MCP 化是过度工程。

## 五、下一步建议

**走阶段 B（度量底座）**：把 `tests/eval` 从骨架激活为真评估——
1. 把 10 个 case 的 `@skip+pass` 实现为真实跑（注入 db + httpx client + LLM-as-judge / 规则判定）
2. 新增**检索评估集**：标注一批"问题 → 应召回的 KP/note doc_id"，实现 `Recall@K / MRR / nDCG`
3. 一条命令出评测报告，作为后续 C 检索升级的 A/B 基线

投入小、可 TDD、是后续一切优化的度量前提。完成后即可有据地推进 C 混合检索。
